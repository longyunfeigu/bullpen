## ② 复核官 · REVIEW_READY 门前的第二意见 ⭐⭐

**一句话**:任务完成后、进入人工审查前,一个独立上下文、只读、可用更强模型的"第二方"先对抗式核一遍——声明 vs 证据逐条对账、覆盖缺口、恒真断言,并把整份 Changes 按风险排好序;证伪即拦下 Full 模式的自动接受。

**为什么(用户视角)**:"agent 说 3 个测试全绿——但我被坑过,它谎报过成功。现在我默认不信,验证工作自己重做一遍。"每任务 1 次、每天 3-5 次,审查是一天里最累的环节。

**为什么(PM 视角)**:直击头号信任缺口(Sonar:96% 开发者不敢信 AI 代码、仅 48% 每次验证;官方 issue 实锤 agent 伪报测试通过)。信任=留存。

**竞品证据**:
- 正面:Amp「Oracle」(独立上下文更强模型出二次意见,https://ampcode.com/manual);Claude Code 高手标配 subagent 复核。
- 反面(设计护栏):CodeRabbit 式逐条 PR 评论 60-80% 是噪音、70-90% 被忽略、合并时长 6h→2-3 天(https://dev.to/synthaicode_commander/why-80-of-ai-code-reviews-are-just-noise-4i0o 、https://news.ycombinator.com/item?id=46766961)——所以形态必须是**门控+风险分级+去重+通过即安静**,绝不做评论机器人。

**与 Charter 资产的接驳**:复核官 = 审查门内的一个只读步骤,经 Tool Gateway 天然受限;发现写入既有证据台账、进 Replay receipt;Full 模式下证伪 = 拦截自动接受回退人工(强化既有"验证失败回退");风险排序复用发现结果,是审查台的 triage 层("先看哪 5%"),与既有 hunk 级接受("怎么接受")正交。与①联动:已沉淀的项目规则可作为复核检查项("违反规则 #13"类发现)。

**界面(见 mock #f2)**:Review tab 顶部复核官条(模型/独立上下文/耗时/成本)→ 高/中/低分级发现卡(每条带证据引用 + Request fix/忽略)→ "其余 N 项声明证据齐备"静默绿区 → Changes 按风险排序 + "只看高风险"过滤 + 低风险批量接受。

**待确认**:默认档位(跟随任务模型力度 high vs 更强模型)?挑剔度默认(只报中高)?成本敏感用户可整体关闭?

## ③ 成本账本 · 每任务归因 + 模型右配 ⭐⭐

**一句话**:每个任务收尾直接给"这单 $0.42、39% 烧在返工",按阶段(计划/编辑/验证/返工)拆分,基于近期同类任务给一句模型右配建议;Home 常驻本周用量小卡 + 预算上限。

**为什么(用户视角)**:"月底账单比我估的高一截,钱烧在哪说不清——/cost 是事后,第三方 ccusage 对不上任务。"每任务收尾看一眼 + 每天一次"这月花了多少"。

**为什么(PM 视角)**:RedMonk 把成本可见列为 table-stakes,2025"定价 rug pull"是明确 churn 驱动(https://redmonk.com/kholterhoff/2025/12/22/10-things-developers-want-from-their-agentic-ides-in-2025/);Kiro 已展示每 prompt 成本;ccusage 的流行即需求实锤(https://agenticcontrolplane.com/blog/claude-code-cost-tracking)。

**结构性独家**:session 型工具只能给"这次会话花了多少";Charter 的任务边界+阶段是状态机既有事实,per-task/per-phase 归因是白捡的;BYO-API + 零遥测 = "对成本像对隐私一样诚实",卖 token 差价的竞品没动力做。

**接驳**:usage 数据 pi runtime 已回传(real-gateway 测试断言过 usage/cost);阶段归属来自任务事件时间线;成本入 Replay receipt;右配建议与⑥配方联动("存为配方默认")。

**界面(见 mock #f3)**:收尾卡成本行 + 阶段拆分条 + 右配建议;Home 周视图小卡($/预算进度/按模型/按项目)+ 预算 modal(超限仅提醒 / 暂停派发)。

**待确认**:定价表内置随版本更新,自定义网关未知价格退化为 token 数——可接受?预算超限默认动作?

## ④ 范围小票 · 越界与依赖护栏

**一句话**:收工时把最终 diff 对照已批计划做**确定性核对**(零模型成本、每任务免费常开):计划内安静放行,计划外单独成组,新增依赖必须显式点头,疑似重造轮子给 nudge。

**为什么(用户视角)**:"我只让它修登录 bug,审到一半发现它顺手重构了 3 个没让碰的文件、手搓了项目里已有的日期工具、还塞了个新依赖。"每任务 1 次"我得确认它没越界"的心智税;真抓到每天 1-2 次。

**证据**:"almost-right / 越改越多"是 66% 用户头号痛点(https://www.faros.ai/blog/best-ai-coding-agents-2026);AI 时代 PR 膨胀 18%、事故 +24%(https://www.builder.io/blog/developers-drowning-in-ai-prs);"宁可重造轮子也不用现成库"(https://blog.tedivm.com/guides/2026/03/beyond-the-vibes-coding-assistants-and-agents/)。

**接驳**:计划审批门管"开工前批不批";小票管"收工后是不是只干了批的那些"——计划门的闭合校验,复用既有计划结构与变更台账;依赖行接 R0-R4 权限引擎既有语义。与②互补(②是 LLM 对抗复核,④是确定性核对),可各自独立采纳。

**界面(见 mock #f4)**:Review 顶部小票卡:✓ 计划内 N 处 / ⚠ 计划外(折叠展开,逐条 保留/否决/问它为什么)/ 📦 新增依赖(允许/否决)/ 💡 疑似重造(对比)。

**待确认**:计划外改动默认强度(折叠展示 vs 阻断 accept 需逐条处理)?重造检测 v1 轻启发式(同名/相似签名)误报可接受?

## ⑤ 恢复条 · 上下文切换复位

**一句话**:切回搁置任务的那一刻,顶部一行人话复位:"你离开 38 分钟,agent 改完 6 文件验证通过等你审查;你上次停在 Diff hunk #3 没做决定"+ 一键跳回。

**为什么(用户视角)**:"处理完 C 回到 A,愣三秒:A 到底跑到哪了?"每次切换 1 次、每天 15-30 次起。Needs-you 收件箱解决"哪个任务要我",恢复条解决"到了之后我在哪"。

**证据**:多 AI changeset 间切换的认知税(https://addyo.substack.com/p/code-review-in-the-age-of-ai);跨会话上下文重建之痛(O'Reilly)。并行 agent 工具都在做"agent 状态看板",没人做"你自己的状态"。

**接驳**:agent 侧摘要 = 既有任务/变更/验证事件的纯投影(无新数据源);"你停在哪" = 画布滚动位置 + 最后聚焦 hunk/tab,与既有 scrollMemory/draftStore 惯例同类。并行 worktree 越强、切换税越高——这是"敢并行"的配套能力。

**界面(见 mock #f5)**:Room 顶部恢复条(agent 进展一句 + 你的停点一句 + 回到那里/从头看/关闭),点击跳回高亮对应 hunk。

**待确认**:出现阈值(mock 按离开 ≥5 分钟且期间有事件)?"停在哪"粒度要不要含未发送批注草稿提示?

## ⑥ 任务配方 · 发射参数模板

**一句话**:把 composer 的**全套发射配置**(prompt 骨架含 {{变量}} + 模型×力度 + 信任模式 + 文件域 + 验证命令)存成配方,`/配方名` 一键装填;配方存项目内纯文件可随 git 分享。

**为什么(用户视角)**:"同一句'补边界测试,只动 test 文件,跑 vitest,信任 Edit,Sonnet 中力度'我这周敲了第四遍——prompt 片段库带不走发射参数。"每天派 5-15 个任务,大半是重复任务型。

**证据**:Sourcegraph Prompt Library(https://sourcegraph.com/blog/announcing-prompt-library);Android Studio prompt library 带模板变量随项目 git 共享(https://developer.android.com/studio/gemini/prompt-library)。但没有工具把发射参数一起模板化——这些维度恰是 Charter composer 已有的。

**与技能的边界**:技能 = agent 能调用的能力;配方 = 你发射任务的参数模板。"/" 面板两个分组并列,心智清晰。

**接驳**:composer 全部字段已存在,配方即其序列化;与③右配建议联动("存为配方默认");与高级 charter(标题/边界/成功标准/验证)是"可复用预填"关系。

**界面(见 mock #f6)**:"/" 菜单配方分组(名称+参数摘要)→ 选中装填 composer({{module}} 高亮占位)→ "存为配方" modal(勾选保存哪些参数)。

**待确认**:配方可否锁定字段(团队规范场景)?{{变量}} 未填时发射前强校验?

## ⑦ 选区改写 · Editor 内 ⌘I

**一句话**:Monaco 里选中几行 → 一句话指令 → inline diff → 回车接受;走既有快照/原子写/字节级回滚管线,连三行小改都可逆、可溯源。

**为什么(PM 视角,top3)**:Cursor 三面(Tab/Cmd-K/Agent)中 Cmd-K 是"只处理选区所以快"的日常高频肌肉记忆(https://www.builder.io/blog/cursor-tips);Karpathy 称为"自主度滑杆"——agent 管跨文件,这档管手边三五行。Charter 的 Editor 表面目前没有任何 AI 触点,用户在这里断流。

**为什么不是幽灵补全**:补全是低延迟军备赛(Cursor 自研 Fusion 日均 4 亿次请求),BYO-API 结构性打不赢——选区改写延迟容忍,复用 agent 基建,是同一高频场景的可赢切片。

**接驳**:复用 agent + 编辑管线(快照/原子写/回滚/权限引擎)——比 Cursor Cmd-K 多出审计与可逆属性;可作为轻量变更计入任务历史,Replay 可溯源。

**界面(见 mock #f7)**:选区下浮出改写条(⌘I,输入指令)→ inline diff(del/ins)→ 接受 ⏎ / 重试 / Esc;角注"经快照 · 字节级可回滚"。

**待确认**:键位——⌘K 已被全局面板占用,mock 按 ⌘I;另一选项"有选区时 ⌘K=改写"会不会歧义?改写记账:轻量变更入任务历史(可回放)vs 只进编辑器撤销栈?

## 附:两视角共识否决清单(不做,防"看起来有用实则没人用")

1. **幽灵补全军备赛** — Cursor Tab 日均 4 亿次靠自研低延迟模型(https://blog.bytebytego.com/p/how-cursor-serves-billions-of-ai);BYO-API 打不赢,做了必然二流。
2. **语音输入** — PM 提议、重度用户实证否决(Cursor 加了又删;"我干嘛对着终端说话")。两视角冲突即弃。
3. **云端后台 agent** — 需自家云,违背单机+零遥测定位;社区口碑"completely unusable"(https://forum.cursor.com/t/cursor-background-agents-completely-unusable/154103)。
4. **自动 PR 评论机器人** — 60-80% 噪音、70-90% 被忽略;正确形态是门控复核官(②)。
5. **团队协作后端 / 排行榜** — 需自建服务端;对单机开发者是伪激励。
6. **MCP 目录 / 市场** — 装十几个两周后只剩 3-4 个常用(https://newsletter.pragmaticengineer.com/p/mcp-deepdive)。
7. **重量级 spec 三件套** — Fowler 实测小任务太啰嗦且 agent 不遵守(https://martinfowler.com/articles/exploring-gen-ai/sdd-3-tools.html);保持轻量计划门。
8. **代码库语义索引** — PM 单方提名但规模 M-L 且用户侧无高频感知诉求;列为观察项,本批不做。
