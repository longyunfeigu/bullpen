方案 C 要点

本质

  • 经 SSH 往远端部署一个 headless 的 agent-host（复用现有 agent worker + Tool Gateway），本地只是 UI
  • 「远程目录」成为一等 project：task / Room / 终端 / 文件操作全部原生作用于远端
  • SSH 只是引导 + 隧道层；产品协议跑在隧道里（版本化，同现有 IPC 契约思路）

优点 / 缺点

  • 与产品定位深度契合：不是「连上服务器自己敲命令」，而是「让 agent 在服务器上干活」
  • 审查门 / 权限门 / 快照回滚在远端照常生效 —— 这是 Termius/Warp 都没有的差异化能力
  • 大文件 / 大 repo 不用拉到本地
  • 工程量最大（数周）：远端二进制分发、版本匹配、断线恢复、平台矩阵
  • 远端环境要求（glibc / 架构）带来支持成本
  • 对「只想快速 ssh 上去看看」的场景太重，仍需要 A 兜底

适合谁

  • 开发 / 调试主要发生在服务器上的用户（GPU 盒子、staging、生产排障）
  • 希望把 Charter 的 agent 工作流带到远端的用户
  • 定位是长期路线：以 A 为地基分阶段实现（先远程终端 → 远程只读 → 完整远程 Room）

VS Code Remote-SSH 的架构（本地 UI + 远端 server，版本严格匹配，经 SSH 隧道通信）。对 Charter 而言额外的红利是：agent 运行时本来就与 UI 进程隔离、走版本化契约，把它放到网络另一端在架构上是自然延伸，而不是重构。
