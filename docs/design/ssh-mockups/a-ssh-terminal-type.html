方案 A 要点

本质

  • 用现有 node-pty 直接 spawn 系统 ssh（host-owned executable，不往 shell 打字）
  • launch 联合类型加 'ssh'，SSH 会话就是一个普通终端 Session
  • 主机清单只读解析 ~/.ssh/config，产品不另存一份

优点 / 缺点

  • OpenSSH 全兼容：密钥、agent、ProxyJump、ControlMaster 零成本继承
  • 不引入新依赖、不碰凭据存储，安全面不变
  • 改动集中在 3 处扩展 + 1 个 config 解析器，1–2 天量级
  • 无 SFTP / 文件传输、无端口转发管理 UI
  • 密码登录体验 = 在终端里敲密码（无保存）

适合谁

  • 已经维护 ~/.ssh/config 的用户（键盘型用户）
  • 把「连上去干活」当核心诉求，而非「管理一批服务器」
  • 是 B / C 两方案的公共地基：先上 A 不会白做

Warp / iTerm2 走的就是这条路线：终端只负责把你送上去。落点：packages/terminal-service 的 CreateTerminalOptions.executable 已支持直接 spawn，UI 只需扩展 NewTerminalDialog 的类型网格。
