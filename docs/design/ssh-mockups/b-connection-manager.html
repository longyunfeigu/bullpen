方案 B 要点

本质

  • 引入纯 JS 的 ssh2 库做协议层，一条连接多路复用 shell / SFTP / 转发
  • 「主机」成为产品一等实体：主机库、标签、密钥引用、最近连接
  • 非敏感配置进 settings.json 新 ssh section；密码 / passphrase 进 keychain（SecretService）

优点 / 缺点

  • SFTP 拖拽传文件、端口转发 UI、断线重连 —— 覆盖运维日常全场景
  • 可编程控制连接（未来 agent 可复用同一条连接跑远程命令）
  • electerm / SimpleShell 均验证过 node-pty + ssh2 混合架构可行
  • ssh_config 兼容要自己做（ProxyJump/agent 是重灾区）
  • 凭据管理引入新安全面，需要 ADR + 安全测试
  • 工程量 1–2 周量级，UI 面积大

适合谁

  • 管理多台服务器、常传文件、常开隧道的用户
  • 希望摆脱 Termius 订阅、把 SSH 工作流收进 Charter 的用户
  • 可分期：先主机库+终端（复用 A 的 spawn），SFTP / 转发二期

Termius / electerm / Tabby 的路线。注意与产品安全架构的契合点：ssh2 连接由主进程持有，renderer 只经由既有版本化 IPC 通道收发数据，SFTP 写本地文件走既有 Tool Gateway 式白名单目录——不新开任意文件读写口子。
