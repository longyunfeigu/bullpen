① 概念：导播位——它解决融合稿的什么盲区

  • 概念：编队区从"均权监视墙"升级为"导播台"。墙（双列瓦片）之上加一块大画面，系统按显式优先级把"当前最值得看"的一路切上来：待你审批 > 失败/异常退出 > 刚完成 > 输出中 > 静默。每次切换必带原因 chip（"切入 · 待审批 R3"），可锁定、可切手动、每次切换留档可回看。
  • 盲区一：小瓦片看不清。融合稿瓦片 10.5px mono 只露三行尾巴；想看清就得「打开房间」——但那会离开指挥位，丢掉主会话叙事和另一路动静。导播位 = 不离开指挥位的深围观。
  • 盲区二：注意力调度靠人肉。两路还能扫，三四路时用户变成轮询器。系统本来就从账本事件里知道该看谁（审批、exit 码、完成判定），导播用一张少而硬的规则表替用户切，而不是只给边框变色。
  • 盲区三：低头即错过。瓦片只显示"现在的尾巴"，你回消息的十秒里 test 窗从 exit 1 走到申请注入的过程就丢了。「导播记录」把每个切换时刻留成快照 chip，点开回看——错过的瞬间兜得住。

② 与融合稿的关系：叠加层，账本的实时投影

  • 纯叠加，不推翻：rail 家谱缩进（子行 44px）、needs 数字冒泡、⌁ 横带、两档暂停、敲字即接管——全部原样。双列瓦片墙（已拍板）保留为"监视墙"，导播位插在墙上方，同属可折叠编队区；折叠后仍只剩一行区头。
  • 审批多了第四入口：rail 徽章 / 瓦片 / 导播位卡 / 被指挥房间卡，四处引用 permission_requests 里同一条记录，任意一处处理，其余同步消失（本页可验证）。被放大的瓦片正文让位为「已在导播位」，未上屏时瓦片保留完整审批按钮——同一记录按表面状态渲染，不是四份状态。
  • 不造新机器：切换由账本里已有的编排事件驱动（审批、exit 码、静默判定），导播记录就是账本的筛选视图，"回看"就是 replay 的 UI 形态——概念落在既有事件账本骨架上。

③ 代价与风险

  • 垂直空间：导播位挤压主会话叙事区。对策：瓦片压矮（尾巴两行，细节归导播位）、整区可折叠；窄高度下可退回"仅墙"形态。
  • 自动切换的失控感：画面自己动是最容易招烦的交互。对策：切换必带原因 chip、一键锁定、手动档（点瓦片头上屏）、审批未决时其他事件只排队不抢画面、同路事件只换 chip 不闪切。规则表必须少而硬——一旦错判几次，用户会永久忽略导播。
  • 回看的实现成本：切出时刻快照需要截取终端 scrollback 并留存（大小、隐私、脏数据）；若走账本 replay 全重建则成本再上一个量级。
  • 规则僵化风险：优先级表不做机器猜测（刻意），但"输出中谁优先"在多路时仍可能不合预期——先用"最久未上屏优先"，观察后调。

④ 留给产品负责人拍的三个问题

  • Q1 导播位要不要全局化？本稿只放在主会话房间（编队区内）。是否要跟随到被指挥房间、甚至浮出壳层跨房间常驻（那就长成画中画的形态）？建议 v1 只留在指挥位，观察使用再说。
  • Q2 回看的保真度选哪档？轻量文本快照（切出时刻的 scrollback 尾巴，本稿演示的形态）vs 基于账本 replay 的完整重建（可拖时间轴）。成本差一个量级，建议 v1 快照、replay 留给账本查看器。
  • Q3 默认档位？自动导播默认开（新手直觉"它自己会切"，但有失控感风险）还是默认手动（保守，导播记录仍然工作）？本稿默认自动 + 显眼的锁定/手动出口。
