融合后的分工（谁负责什么）

  • rail = 到达与全局感知：家谱缩进（沿用组内缩进习语）回答"谁指挥谁"；子会话的 needs 冒泡到主会话行，Needs 过滤器一键聚焦。
  • 主会话房间 = 围观与快处理：编队区内嵌实时瓦片，审批可直接在瓦片上批掉，不必离开指挥位；区块可折叠（折叠态记忆），不围观时房间回到纯对话形态。
  • 被指挥房间 = 深入与接管：⌁ 横带写明"被谁指挥"（点名字即回主会话）；点终端即接管，注入暂挂；审批卡在此同样出现。
  • 层级只到两层（主会话 → 工作会话）：子会话再开子会话在 v1 不允许（工程与心智都简单一个量级）。

一个事件、三个入口（本页可验证）

  • 那条 R3 注入请求同时呈现为：rail 的「待审批」徽章（+主会话行的 needs 计数）、主会话房间瓦片上的按钮、被指挥房间的内联卡。
  • 在任意一处允许/拒绝，其余入口同步消失——UI 三处引用的是 permission_requests 里的同一条记录，不是三份状态。系统通知（PIVOT-014）是第四入口，本 mock 不演示。
  • 允许后可观察完整余波：t9 终端重装→测试转绿 exit 0→主会话被 wait 唤醒→瓦片徽章变「exit 0」→rail 圆点复绿。

交互契约细则

  • 敲字即接管：被指挥终端的任何本地输入使遥控注入进入暂挂队列；瓦片同步显示「✋ 已接管」；归还后暂挂依次放行。
  • ⏸ 暂停全部遥控：只拦"新注入"，不打断在跑的命令、不关窗口；再点恢复。徽章在 rail 与瓦片同步。
  • 编队区位置：置于终端下方——主会话的叙事优先，围观是从属；折叠后仅剩一行区头。
  • 命名沿用现有惯例：会话名 =「CLI 名 · 意图」；关系词统一用「指挥 / 由…指挥」；全部界面无 Pi 字样。
