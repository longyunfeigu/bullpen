关键洞察：终端不是一直是「agent 终端」——它是一个 shell，agent 会话在里面 开始和结束。敲 claude / codex 的瞬间，PTY 从「命令行」 变成一个竖排的聊天式 TUI（欢迎横幅、输入框、流式回答）。所以「放哪」的答案 取决于「里面正在跑什么」：shell 是横的（矮宽舒服），chat TUI 是竖的 （高窄舒服）。三个方向按这个矛盾展开。检测手段：PTY 子进程名 / OSC 标题变化 / alternate-screen 切换——三者都能可靠感知「进入了交互态」。

无论选哪个方向，都要回答的细节

这些决策与位置无关，三个方向共用。

怎么知道「进入了交互态」

三个信号按可靠度排：① PTY 前台子进程名（claude/codex，node-pty 可查）；② 进程切换到 alternate screen / 改写 OSC 0 标题；③ 输出特征（欢迎横幅）。①+② 组合即可驱动 tab 改名、EXT 徽章、升格提示。

键盘焦点边界

TUI 需要吃掉几乎所有按键（Esc 是打断 agent、⌘K 可能是它的快捷键）。终端聚焦时显示焦点描边，保留一个不可覆盖的跳出键（如 ⌘⇧L），全局快捷键表在终端聚焦态自动收缩到白名单。

尺寸底线

Claude Code / Codex 的 TUI 舒适高度 ≳ 20 行、宽度 ≳ 80 列。右侧竖栏默认 340–420px（够 80 列的 11px 等宽）；底部坞若容纳 TUI 至少要能一键「拉高到 60%」。

多会话并存

右侧栏 tab 化：✳ claude 和 ◆ codex 并列，各自带活动呼吸点；底部坞继续放普通 shell。agent tab 空闲 N 分钟后淡化，会话结束变成「已结束 · 查看改动」入口。

与上一稿（A–D）的关系

位置 × 集成深度是正交矩阵：比如「右侧竖栏 + FS watcher 改动提升（A）」或「升格为房间（= 上一稿方向 A 的完全体）」。本稿的「升级为房间」按钮就是两稿的接缝。

安全语义不变

不管终端在哪，外部 CLI 都在 Tool Gateway 之外：进入交互态即挂 EXT 徽章、自动 snapshot、FS watcher 开始记账——这三件事跟着「检测到 agent」信号走，而不是跟着位置走。

对比与推荐

方向 shell 体验 agent TUI 体验 实现成本 主要风险
1 · 底部坞 最好（形状对） 差（被压扁） 最低 用户长期在窘迫的 TUI 里干活
2 · 右侧竖栏 差（窄栏折行） 最好（形状对） 低 shell 场景别扭；挤占编辑器宽度
3 · 检测升格 最好 最好 中（检测 + 迁移动效） 自动移动需驯化：动效交代 + 首次询问
推荐：方向 3 作为目标，1 + 2 是它的两个静态退化形态。
实施顺序：先落底部坞（方向 1，基础设施）+ 右侧竖栏容器（方向 2，纯布局）；再加检测信号 （前台进程名 + alternate screen），把「移到右侧」先做成 toast 里的一键动作（方向 1 演示的样子）——不自动移动，先收集使用习惯；确认大家总是点它之后，再把默认值翻成自动升格 （方向 3），并保留「总是问我 / 总是移 / 从不移」设置。「升级为房间」按钮始终存在， 通往上一稿的方向 A。
