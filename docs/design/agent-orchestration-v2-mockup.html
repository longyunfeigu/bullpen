三个变体到底哪里"本质"不同

① 家谱在侧栏② 指挥室③ 编排即工具流
子会话是什么一等公民的会话，只是有了父子关系主指挥房间的"舷窗"——编队瓦片一次长时工具调用——时间线里的证据块
感知中心rail（rail 本身就是指挥树）主指挥的房间（一处看全编队）对话叙事（读故事顺带看现场）
rail 变化长出缩进层级 + needs 冒泡不动，仅加 ⌁ 记号子会话不上 rail（被任务收编）
房间变化被指挥房间加一条 ⌁ 横带主指挥房间长出编队区时间线多 ⌁ 条目 + 内嵌实时块
最适配的司机外部 CLI（claude/codex 互相指挥）外部 CLI托管 agent（有产品时间线可长）
主要代价rail 行进一步变挤；层级与项目分组要商量共存房间变复杂；围观和对话抢同一屏"实时围观感"最弱；外部 CLI 当司机时没有时间线可挂

全变体共享的不变量（不随选型变）

  • 审批三入口同一事件：mission control「Needs you」卡、房间内联卡、系统通知——全部走 PIVOT-013/014 既有管道；三处任一处理，其余同步消失。
  • 敲字即接管：在被指挥终端输入，遥控注入自动暂挂（变体一里点终端正文可试）。人永远赢过 agent。
  • ⌁ 事件进账本：开窗/注入/等到 exit 码/关闭全部入 tool_calls，replay 免费回放；变体三的时间线渲染即是 replay 的实时形态。
  • 全部界面无 Pi 字样；会话名沿用「CLI 名 · 意图」惯例。

我的倾向与请你拍的两个开关

  • 三者可组合：①为骨（rail 是家谱）+ ③免费送（托管 agent 当司机时时间线天然长这样）+ ②降级为主指挥房间里可折叠的编队区。
  • 开关一：子会话上不上 rail？上（①②）= 每个子会话可直达、可单独围观；不上（③）= rail 干净，但子会话只能从主会话进。
  • 开关二：rail 长不长层级？长（①）= 指挥关系一眼可见；不长（②③）= rail 结构不动，关系靠 ⌁ 记号与房间表达。
  • 看的时候请重点感受：哪个变体让你最不需要学新东西就明白"谁在指挥谁、谁需要我"。
