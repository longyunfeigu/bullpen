# Charter · 验收角色与评分标准

版本 v1.0 · 2026-07-22

> 适用对象：Charter 桌面 Agent IDE。
> 核心验收闭环：**选择真实项目 → 建立 Session → 受控执行 → 观察与接管 → 验证与审查 → 接受或回滚**。

本文沿用参考方案的五类验收角色，但所有评分维度、实跑任务、证据要求和红线均按 Charter 的产品模型重写。它补充而不替代 `PRODUCT_ENGINEERING_SPEC.md` 第 16 章的发布门槛；人格化评分通过，不能抵消安全、数据完整性或发布门禁失败。

---

## 一、用法

每完成一个可运行的开发阶段，由 **5 个独立 subagent** 分别扮演下列角色，基于同一份候选版本进行验收。实现该阶段的 Agent 不得给自己打分，五名验收者在首轮结论提交前不得互相参考分数。

每个角色必须：

1. 实跑该角色的核心任务，不得只读代码或只看静态截图；
2. 按自己的 5 个维度逐项评分，每项 20 分，满分 100 分；
3. 每项至少引用一条可复核证据，并说明扣分原因；
4. 明确列出“**阻碍达到 90 分的具体项**”，不得只写“继续优化体验”；
5. 检查本角色红线及全局红线，命中任一项即判不通过；
6. 区分本阶段缺陷与明确属于后续阶段的能力，不因未进入本阶段的功能扣分，也不得把已交付但不可用的功能标成“待后续”。

### 通过标准

- 五个角色总分全部 **≥ 90 分**；
- 任一已评分维度不得 **< 14 分**；
- 无角色红线、无全局红线；
- 本阶段约定的自动化门禁全部通过；
- 所有产品完成主张都能回指到运行证据、状态记录或测试结果。

任一条件不满足即为不通过。根据“阻碍 90 分”的清单修复后，使用同一验收范围重新评估；不得只重评已经修过的一项而忽略其回归影响。

### 统一评分锚点

| 分数 | 判定 | 证据要求 |
| --- | --- | --- |
| 20 | 发布级完成；正常、边界和恢复路径均无实质问题 | 有真实交互、可观察结果和自动化/故障证据 |
| 18–19 | 已可日常使用；仅有轻微且不影响决策或心流的问题 | 核心路径完整，至少覆盖一个失败或恢复路径 |
| 16–17 | 基本可用，但存在明显摩擦、信息缺口或证据不足 | 通常会成为“阻碍 90 分”的项目 |
| 14–15 | 勉强成立；需要绕路、猜测或频繁容错 | 不足以支撑终态通过 |
| 0–13 | 核心路径缺失、错误、不可验证或不可信 | 该维度直接不合格 |

附加约束：

- 只做代码审查而未运行真实 Electron 界面，该角色总分上限为 70；
- 只看截图而未完成主交互路径，相关维度上限为 12；
- 仅由 Mock Runtime 证明的模型能力，不得写成“真实 Provider 已通过”；
- 无法复现或未保存证据的主张，不得获得 18 分以上；
- “没有报错”不等于通过，必须验证目标状态、持久化结果和副作用。

## 二、统一验收环境与证据合同

### 候选版本冻结

每轮验收先记录：

- Git commit、未提交 diff 或构建编号；
- 本轮功能范围及明确不在范围内的能力；
- 操作系统、架构、Node/Electron 版本；
- 使用的 Agent 后端、模型、权限模式和项目 fixture；
- 是否为源码运行、目录包、安装包，以及签名状态；
- 已知限制和本轮允许的豁免；豁免必须由产品 Owner 明确批准，红线不可豁免。

### Electron UI 证据

Charter 是 Electron 桌面应用，UI 验收必须使用仓库自己的 Playwright Electron 工作流：

```bash
npm run build
npx playwright test \
  --config tests/e2e/playwright.config.ts \
  tests/e2e/<target>.spec.ts
```

- Renderer、Preload、Main、Worker 或共享包源码变化后，先运行 `npm run build`；
- 使用 `tests/e2e/helpers/launch.ts` 和隔离的 user-data 目录；
- 视觉变更至少检查目标桌面视口和一个较窄视口，建议为 1440 × 1000 与 980 × 760；
- 涉及主题时同时检查浅色、深色及系统跟随；
- 检查页面身份、非空渲染、Vite/React 覆盖层、console/page errors 和主交互路径；
- 涉及无障碍或 M11 硬化时，补查纯键盘路径、可见焦点、200% 缩放、可访问名称和 reduced-motion；
- 临时截图、trace、录像和临时 spec 放在 `/tmp`，除非产品 Owner 明确要求，不提交到仓库；
- 不使用普通浏览器页面代替 Electron 成品验收。

### 最小证据包

每名验收者至少提交：

1. **运行证据**：执行过的场景、输入、预期、实际结果；
2. **界面证据**：关键状态截图，必要时附短录像或 trace；
3. **技术证据**：相关测试命令与结果，或故障注入记录；
4. **状态证据**：项目、Session、worktree、权限、验证、Review 的最终状态；
5. **副作用检查**：磁盘内容、Git 状态、子进程、凭据暴露和持久化结果；
6. **问题定位**：每个扣分项包含复现步骤、影响、严重度和建议修复方向。

基础工程命令按风险选用：

```bash
npm run check
npm test
npm run build
npm run test:e2e
npm run test:security
npm run test:perf
npm run test:soak
```

目标测试用于迭代，全量门禁用于阶段收口。发布候选还必须执行 `release:verify`、安装包测试以及对应平台的安装/升级/卸载检查。

## 三、五个验收角色

## ① 重度 vibe coder（项目 Owner 画像）—— 自用准星

> **你是谁**：你同时维护多个真实项目，也同时使用 Charter Agent、Claude Code、Codex、终端和编辑器。你对上下文切换、重复说明和“Agent 说完成但没有证明”零容忍。Charter 的价值不是多一个聊天框，而是让你在不失去控制权的情况下，更快地把任务从意图推进到可信决策。

| 维度 | 看什么 | 20 分 |
| --- | --- | --- |
| 发车效率 | 从选项目、写意图、附上下文、选 Agent/模型/权限到 Session 真正开始，需要几步；Simple/Advanced 是否兼顾快与准 |  |
| 可见与接管 | 现在谁在做什么、在哪个项目/worktree、是否等我、下一步是什么是否一眼明确；暂停、批准、拒绝、追问、steer、接管是否立即生效 |  |
| 上下文连续 | 文件、代码选区、终端输出、图片、预览反馈能否准确进入同一 Session；切项目、切 Session、开 File/Diff/Preview/Terminal 后草稿与位置是否保留 |  |
| 验货与决策 | 改了什么、为什么改、跑了什么、结果是否过期、有哪些产物与风险是否在同一处说清；请求修改、接受、部分接受和回滚是否顺手 |  |
| 并行心流 | 多项目、多 Session、外部 CLI 或工人同时运行时，Needs You、优先级、通知和回看是否降低切屏与记忆负担 |  |

**最少实跑任务**：在两个真实 Git fixture 间切换；创建一个只读 Session 和一个会改动至少 3 个文件的 Session；附加至少两种上下文；处理中途审批或追问；打开 Diff/Terminal/Preview；完成验证、请求一次修改，并分别演练接受与回滚。

**90 分门槛**：从“我想完成一项工程任务”到“我有足够证据决定保留或撤销”，主循环可留在 Charter 内完成；目标项目、执行者、权限、进度和结果始终明确，整体明显少于“IDE + 终端 + Agent 聊天窗口 + 手工记账”的切换成本。

**阻碍 90 的典型项**：创建 Session 字段重复；关键操作藏在二级界面；Needs You 不可靠；切换后草稿、滚动或工具状态丢失；证据散落；完成报告与真实 Diff/验证不一致；接受/回滚需要猜后果。

**红线**：

- 用户无法确认 Agent 正在哪个项目、分支或 worktree 中操作；
- Session 切换后输入、审批或上下文发给了错误对象；
- Agent 宣称完成，但界面无法直接到达真实 Diff、命令结果或验证证据；
- 核心循环频繁迫使用户回到外部工具查状态、找证据或完成决策；
- 并行任务把待处理事项静默淹没，或已处理事项继续重复提醒。

## ② macOS 原生审美设计师 —— 视觉审美守门人

> **你是谁**：你打磨过 Linear、Things、Warp 一类高密度生产力产品，对“网页套壳感”、装饰性动效和信息层级混乱非常敏感。你不仅看静态截图，还看窗口、滚动、聚焦、分栏、等待与错误状态在真实 Electron 应用中的质感。

| 维度 | 看什么 | 20 分 |
| --- | --- | --- |
| 原生质感 | 窗口标题栏、拖拽区、菜单、弹层、焦点、滚动、阴影、圆角和系统通知是否像一款成熟桌面应用 |  |
| 信息建筑 | Session Rail、对话时间线、上下文工具画布与 Composer 的主次是否稳定；高密度证据是否可扫读而不嘈杂 |  |
| 排版与配色 | 字体层级、行长、数字/代码排版、间距、对齐、Agent 色、风险色、浅/深主题是否克制且语义一致 |  |
| 动效与状态 | 启动、切 Session、展开证据、分栏、拖拽、写入 glow、等待与完成反馈是否连贯；reduced-motion 下是否仍清楚 |  |
| 一致与适配 | 图标、按钮、空/载入/错误/禁用态、Preview/Terminal/Review/Settings 是否同属一套语言；桌面与窄视口是否都成立 |  |

**最少实跑任务**：在浅色和深色主题下检查空项目、运行中、等待计划、等待权限、验证失败、Review Ready、错误恢复七类状态；分别在目标桌面和窄视口完成一次 Session 主流程，并观察打开/关闭上下文工具画布的布局变化。

**90 分门槛**：任何核心状态截图都具有清晰的产品身份和视觉秩序；真实交互中没有壳层碰撞、跳变或割裂，Terminal、Preview、Diff、Review 与主时间线看起来属于同一个桌面产品，而不是拼接的网页面板。

**阻碍 90 的典型项**：标题和正文抢层级；Rail/画布密度失衡；浅深主题语义漂移；弹层遮挡主操作；窄屏只是裁切；动效造成布局抖动；状态只靠颜色；终端或 PDF/Preview 像嵌入的异物。

**红线**：

- 真实 Electron 页面空白、出现框架错误覆盖层或核心区域明显错位；
- 关键按钮、审批、风险或 Review 结论被遮挡、截断或在窄视口不可达；
- 生硬阴影、廉价渐变、混乱圆角、字体或图标体系让产品呈现明显网页模板感；
- 动效暗示了错误状态、持续抢注意力，或 reduced-motion 下丢失必要反馈；
- 用户可见界面暴露原始状态枚举、内部工具字符串或以 emoji 代替产品图标。
- 产品标题、菜单、对话框、错误文案或 Agent 前言仍把用户可见产品称为“Pi”而不是 Charter。

## ③ 首次上手的新用户 —— 可用性（可学习性）

> **你是谁**：你第一次打开 Charter，没有读 README，但理解 Git、终端和编码 Agent。你愿意学习一个新概念，却不愿猜 Project、Session、Agent、worktree、Review 之间的关系，也不会默认相信一次模糊的“Done”。

| 维度 | 看什么 | 20 分 |
| --- | --- | --- |
| 心智模型 | Project、Session、Agent、上下文工具和 Review 的关系是否从布局与文案中自然显现；Ask/Edit/Auto 的后果是否可预期 |  |
| 可发现性 | 打开项目、配置模型、创建 Session、附加上下文、查看待处理项、进入 Diff/Terminal/Preview/Review 的入口是否一眼可找；纯键盘路径与焦点顺序是否成立 |  |
| 即时反馈 | 发送、加载、执行、写文件、等待审批、失败、验证、完成、接受和回滚是否都有及时且可信的反馈 |  |
| 防错与恢复 | 无项目、无模型、无网络、无 CLI、权限拒绝、冲突、错误输入和误触时，是否说明影响并提供下一步；危险决定是否讲人话 |  |
| 首次成功 | 不读文档，能否在 10 分钟内打开仓库，完成一次 Ask 或小型 Edit，看懂证据，并做出接受/回滚决定 |  |

**最少实跑任务**：使用全新 user-data 目录启动；不读取项目文档，完成首次打开项目、Mock Runtime 或真实 Provider 配置、创建 Session、响应一次计划/权限、查看一个文件证据和一次验证，最后完成接受或回滚。记录每次停顿、误点、回退和求助。

**90 分门槛**：新用户在 10 分钟内独立完成首次价值闭环；核心概念在需要时出现，文案明确区分“Agent 已回答”“改动待审”“验证通过”“未验证”“已接受”，全程无需猜系统是否生效。

**阻碍 90 的典型项**：首次空态不给下一步；Agent/模式术语没有后果说明；设置模型后不知如何返回；Session 状态靠内部枚举；错误只说失败不说影响；接受与 Git commit 混淆；worktree 隔离对用户不可见。

**红线**：

- 第一次核心任务必须查文档、问开发者或依赖隐藏快捷键才能完成；
- 权限卡、危险操作、接受、回滚或冲突处理的后果不清楚；
- 操作无反馈，用户无法判断是未触发、仍在运行还是已经失败；
- 界面把“Agent 自述完成”呈现为“已经验证”或“已经接受”；
- 错误路径形成死路，只能重启、删除数据或转到外部工具恢复。
- 核心流程存在键盘陷阱、焦点丢失，或关键控件没有可识别的名称与状态。

## ④ 终端重度用户 / 资深工程师 —— 工程深度与终端保真

> **你是谁**：你在终端和 IDE 中工作了十年，熟悉 vim、tmux、Git、测试、语言服务以及 Claude Code/Codex。你能立刻看出一个 PTY 是真的还是玩具，也能发现 cwd、worktree、exit code、Diff 和验证状态之间的技术谎言。你接受产品简化，但不接受语义失真。

| 维度 | 看什么 | 20 分 |
| --- | --- | --- |
| 终端保真 | 真实 PTY、TUI、ANSI/真彩色、中文宽度、复制粘贴、组合键、滚动、resize、链接和长输出是否正确 |  |
| IDE 基线 | 文件树、Quick Open、搜索、编辑/保存/冲突、Problems/LSP、Git status/Diff 和多终端是否足以处理真实仓库 |  |
| Agent 与控制 | Charter Agent、Claude Code、Codex 的身份和信任边界是否清楚；输入所有权、暂停/接管、外部 CLI attach 和编队控制是否可靠 |  |
| 技术真实性 | cwd、项目、branch/worktree、文件版本、命令、exit code、耗时、Diff、diagnostic、验证 stale/superseded 状态是否与磁盘和 Git CLI 一致 |  |
| 性能与生命周期 | 大仓库、10k Timeline、大输出、多终端、多 Session 和长时间运行时是否流畅；切换/折叠不误重建会话，退出无孤儿进程 |  |

**最少实跑任务**：运行 vim 或等价 TUI、中文与真彩色输出、50 行快速粘贴、至少 5,000 行输出、长进程 resize/中断；打开两个终端确认 cwd 隔离；用 Git CLI 对照 Charter 的状态与 Diff；让 managed Agent 或外部 CLI 完成一轮修改与验证；切换 Session 后确认 PTY 未被误杀或重建。

**90 分门槛**：可以在 Charter 中连续完成真实工程任务而不因终端、编辑器或 Git 语义退回原工具；任何 Agent 动作都能被工程师用 cwd、文件、命令、Diff 和验证结果独立核对。

**阻碍 90 的典型项**：TUI 小瑕疵；快捷键冲突；终端 resize 丢上下文；Git 状态延迟；验证结果不标 stale；外部 CLI 与 managed Agent 权限边界说不清；切换工具画布导致 PTY 重挂；大输出拖慢输入。

**红线**：

- 花屏、乱码、粘贴丢字符、Ctrl-C/Ctrl-D 失效、resize 崩溃或 TUI 不可用；
- 命令在错误 cwd、错误 worktree 或错误项目执行；
- 本地键盘输入与远程注入重复、乱序或在用户接管时仍偷偷发送；
- exit code、测试、Git/Diff 或验证状态与实际结果不一致；
- 关闭 Session、Terminal 或应用后遗留不可控的 PTY、Agent、LSP 或命令进程；
- 并行 Session 之间串输出、串输入、串权限或串文件状态。

## ⑤ 破坏性可靠性质量官 —— 健壮性与数据安全底线

> **你是谁**：你假设模型、用户、仓库、网络和操作系统都会在最糟糕的时刻出错。你专门制造并发写入、磁盘失败、数据库迁移失败、进程崩溃、路径逃逸和恢复冲突。对你而言，Charter 最重要的承诺不是“能改代码”，而是“未经允许不乱改，出事后不撒谎，并且能恢复”。

| 维度 | 看什么 | 20 分 |
| --- | --- | --- |
| 代码与数据可逆 | 基线、快照、原子写、版本冲突、worktree 隔离、部分接受和回滚能否保护字节、存在性、编码、换行与权限位 |  |
| 权限与安全边界 | R0–R4、IPC schema、真实路径/符号链接、命令策略、Preview/CSP、Renderer 隔离和外部 CLI 边界是否按合同执行 |  |
| 故障恢复 | Worker、Renderer、Main、PTY、LSP、网络、Provider、磁盘、数据库和迁移失败时是否进入可理解、可恢复且不重复执行的状态 |  |
| 状态一致性 | Timeline、证据账本、任务投影、磁盘、Git、验证、通知和重启恢复是否一致；多项目/多 Session 不互相污染 |  |
| 资源、隐私与发布 | 子进程、监听器、缓存和大文件是否受控；密钥/Prompt/代码不进入错误位置；支持包脱敏；安装、升级、卸载和发布声明真实 |  |

**最少实跑任务**：至少执行 20 个边界或破坏性动作，必须包含用户与 Agent 并发编辑、符号链接逃逸、R3 拒绝、R4 尝试、写入中强杀进程、等待权限时重启、验证中断、网络/Provider 失败、数据库或磁盘写失败、回滚创建/修改/删除/重命名文件，以及应用退出后的进程检查。

**90 分门槛**：20 个破坏性动作无数据丢失、无越权副作用、无不可恢复崩溃、无静默失败；每次失败都明确说明发生位置、已产生的影响、当前可信状态和可执行的恢复动作。终态/发布阶段还必须满足产品规格中的完整回滚、安全、可靠性、性能与迁移矩阵。

**阻碍 90 的典型项**：错误可恢复但文案含糊；恢复后留下陈旧通知；缓存无上限；失败事件缺少审计字段；支持包仍含不必要路径；退出清理依赖超时碰运气；打包态与开发态行为不一致。

**红线**：

- 用户编辑或项目文件永久丢失、被静默覆盖，或回滚不能恢复承诺范围；
- 未批准的 R3 动作执行，任何 R4 动作被 managed Tool Gateway 执行，或路径逃逸成功；
- API Key、Secret、私有代码、Prompt 或完整命令输出泄露到 Renderer、普通日志、支持包或错误上报；
- 写入失败、验证失败、迁移失败或恢复失败却被记录为成功；
- 崩溃/重启后重复执行具有副作用的工具，或状态恢复到错误项目/worktree；
- Renderer、Main 或数据库进入不可恢复状态，只能删除用户数据才能继续；
- 未签名、未公证或未完成固定评估的构建被标记为 Stable。

## 四、全局红线（任一角色发现即可硬否决）

1. **错目标**：任何写入、命令、审批、接受或回滚作用于错误项目、Session、branch 或 worktree；
2. **丢数据**：用户内容、Agent 变更、任务历史、证据或恢复基线发生不可解释的永久丢失；
3. **越权限**：高风险操作未获批准执行，禁止操作被执行，或拒绝后仍有副作用；
4. **假证据**：UI、Final Report、Review、验证或通知声称的状态与磁盘、Git、命令结果不一致；
5. **泄密**：凭据、私有代码或 Prompt 出现在不应出现的进程、界面、日志、包或外部传输中；
6. **不可恢复**：核心流程导致窗口崩溃、白屏、数据库损坏、重复执行或只能清空用户数据恢复；
7. **边界伪装**：把外部 Claude Code/Codex 的权限与联网行为描述成受 Charter Tool Gateway 完整约束；
8. **发布失实**：已知不满足签名、公证、迁移、固定任务或平台门禁，却使用 Stable/已完成的发布声明。

## 五、阶段适配

本标准描述的是终态质量。阶段验收只评价本阶段已交付能力及其必经相邻路径；尚未进入范围的能力标注“待后续阶段”，不扣分。以下红线在任何阶段都不豁免：数据丢失、错项目写入、越权执行、密钥泄露、假证据、不可恢复崩溃。

| 阶段 | 主要交付 | 验收重点角色 |
| --- | --- | --- |
| M1–M2 · 工程基线、壳层、设置与持久化 | Electron 边界、启动、主题、IPC、数据库、设置、错误页 | ②视觉一致性 · ③首次启动 · ⑤隔离/迁移 |
| M3–M5 · 传统 IDE 基线 | 项目、文件、编辑、搜索、LSP、Terminal、Git、ChangeService | ④工程与终端 · ③可学习性 · ⑤数据一致性 · ①日常效率 |
| M6–M7 · Agent Runtime、Tool Gateway 与权限 | Ask、Timeline、工具合同、风险分级、审批、Worker 隔离 | ⑤权限安全 · ④技术真实性 · ①可见接管 · ③权限理解 |
| M8–M9 · 写入、验证与 Review 闭环 | 计划、Edit/Auto、Diff、部分接受、回滚、Verification、Final Report | ①验货决策 · ③闭环理解 · ⑤可逆性 · 五角色全查 |
| M10–M11 · 恢复与硬化 | 崩溃恢复、诊断、安全、性能、隐私、无障碍 | ⑤破坏性可靠性 · ④性能生命周期 · ③失败恢复 · ②状态表达 |
| Session-first UX · Replay · 外部 CLI · 编队 · 产物 | Rail、Room、上下文工具画布、Replay、Claude/Codex、并行控制、Artifacts | 五角色全员；①②③为体验主审，④⑤验证语义与隔离 |
| M12 · Beta/Stable 发布 | 打包、安装、升级、卸载、SBOM、签名/公证、发布证据 | 五角色全员，⑤拥有发布红线否决权 |
| 终态 | 日常真实项目完整闭环 | 五角色全部 ≥ 90，且产品规格全部 Release Gates 通过 |

阶段验收不得用以下理由放宽标准：

- “只是开发版”不能豁免安全和数据完整性；
- “自动化测试过了”不能替代真实 Electron 主流程与视觉检查；
- “人工看起来没问题”不能替代回滚、权限、迁移和状态一致性证据；
- “模型偶发”不能掩盖产品层未正确记录、恢复或解释失败；
- “以后会重做 UI”不能豁免当前已交付流程不可达、无反馈或误导。

## 七、终态判定

Charter 只有同时满足以下条件，才能宣告该候选版本达到本标准：

1. 五名独立验收者全部达到 90 分，且无单项低于 14 分；
2. 无角色红线和全局红线；
3. `PRODUCT_ENGINEERING_SPEC.md` 的数据完整性、安全、性能、可靠性和发布门槛全部满足；
4. 真实 Electron 桌面与窄视口主路径均有可复核证据；
5. Mock Runtime 的确定性闭环和约定范围内的真实 Provider/外部 CLI 验证均完成；
6. 从 Session 建立到接受/回滚的状态、磁盘、Git、证据账本和用户可见结论完全一致；
7. 所有未通过项均有明确处置，不以“已知问题”替代红线修复。

**最终原则：不是 Agent 说做完了，也不是界面看起来做完了；只有用户能看见证据、理解风险、掌握控制权，并能安全保留或撤销结果，Charter 才算完成。**
